# Requirements Document

## Introduction

MindGuard AI is a proactive mental health and burnout prevention companion designed for women navigating the pressures of career and family life. Unlike reactive mental health tools, MindGuard AI continuously monitors emotional patterns through voice and text journaling, applies AI-driven sentiment analysis, and delivers personalized coping strategies and proactive burnout alerts — before a crisis occurs. The system prioritizes user privacy, data security, and a calm, accessible user experience.

---

## Glossary

- **System**: The MindGuard AI application and its backend services collectively.
- **User**: A woman using the MindGuard AI application to track and manage her emotional wellbeing.
- **Journal_Entry**: A voice or text submission made by the User to express emotional state or daily experiences.
- **Sentiment_Analyzer**: The component powered by Amazon Comprehend that detects sentiment and emotion from Journal_Entries.
- **Transcription_Service**: The component powered by Amazon Transcribe that converts voice Journal_Entries to text.
- **AI_Engine**: The component powered by Amazon Bedrock that generates personalized insights, coping suggestions, and burnout predictions.
- **Trend_Store**: The Amazon DynamoDB database that persists User emotional data and behavioral patterns over time.
- **Notification_Service**: The component powered by Amazon SNS and Amazon Pinpoint that delivers smart nudges, alerts, and escalation messages.
- **Processing_Pipeline**: The AWS Lambda-based backend that orchestrates data flow between all components.
- **Burnout_Score**: A numeric value (0–100) computed by the AI_Engine representing the User's current burnout risk level.
- **Trusted_Contact**: A person designated by the User to receive emergency emotional support escalation notifications.
- **Emotional_Health_Report**: A weekly summary of the User's emotional trends, patterns, and AI-generated insights.
- **Coping_Suggestion**: A personalized recommendation (e.g., breathing exercise, break reminder, mindfulness prompt) generated by the AI_Engine.
- **Escalation_Threshold**: A configurable Burnout_Score value above which the System triggers emergency support escalation.

## Requirements

### Requirement 1: Voice Journaling

**User Story:** As a User, I want to record my thoughts and feelings by voice, so that I can express myself naturally without the friction of typing.

#### Acceptance Criteria

1. WHEN a User submits a voice Journal_Entry, THE Transcription_Service SHALL convert the audio to text within 30 seconds.
2. WHEN the Transcription_Service completes transcription, THE Processing_Pipeline SHALL pass the resulting text to the Sentiment_Analyzer.
3. IF the Transcription_Service fails to process a voice Journal_Entry, THEN THE System SHALL notify the User of the failure and retain the original audio for retry.
4. THE System SHALL accept voice Journal_Entries in audio formats including MP3, WAV, and M4A.
5. THE System SHALL support voice Journal_Entries of up to 10 minutes in duration.

---

### Requirement 2: Text Journaling

**User Story:** As a User, I want to write text journal entries, so that I can log my emotional state when voice input is not convenient.

#### Acceptance Criteria

1. WHEN a User submits a text Journal_Entry, THE Processing_Pipeline SHALL pass the text to the Sentiment_Analyzer within 5 seconds.
2. THE System SHALL accept text Journal_Entries of up to 5,000 characters.
3. THE System SHALL store each Journal_Entry in the Trend_Store with a UTC timestamp and the User's identifier.
4. IF a User submits an empty Journal_Entry, THEN THE System SHALL return a validation error and SHALL NOT store the entry.

---

### Requirement 3: Real-Time Sentiment and Emotion Analysis

**User Story:** As a User, I want my journal entries analyzed for sentiment and emotion, so that I receive accurate insights about my emotional state.

#### Acceptance Criteria

1. WHEN a Journal_Entry text is received, THE Sentiment_Analyzer SHALL classify the entry with a sentiment label (POSITIVE, NEGATIVE, NEUTRAL, or MIXED) and a confidence score between 0.0 and 1.0.
2. WHEN a Journal_Entry text is received, THE Sentiment_Analyzer SHALL detect emotion categories including joy, sadness, anger, fear, and disgust with individual confidence scores.
3. THE Processing_Pipeline SHALL store the sentiment label, emotion categories, and confidence scores in the Trend_Store alongside the corresponding Journal_Entry.
4. IF the Sentiment_Analyzer returns an error, THEN THE Processing_Pipeline SHALL log the error, store the raw Journal_Entry without analysis, and retry analysis up to 3 times.

---

### Requirement 4: Burnout Prediction

**User Story:** As a User, I want the system to detect when I am at risk of burnout, so that I can take preventive action before my wellbeing deteriorates.

#### Acceptance Criteria

1. WHEN a new Journal_Entry is analyzed, THE AI_Engine SHALL compute an updated Burnout_Score for the User based on emotional trends from the preceding 30 days stored in the Trend_Store.
2. THE AI_Engine SHALL factor in frequency of negative sentiment, intensity of detected stress and fatigue emotions, and journaling gaps of more than 5 consecutive days when computing the Burnout_Score.
3. THE System SHALL store each computed Burnout_Score in the Trend_Store with a UTC timestamp.
4. WHEN the Burnout_Score exceeds 70, THE Notification_Service SHALL send the User a proactive burnout risk alert.
5. THE AI_Engine SHALL recompute the Burnout_Score at least once every 24 hours for active Users, independent of new Journal_Entries.

---

### Requirement 5: Personalized Coping Suggestions

**User Story:** As a User, I want to receive personalized coping suggestions based on my emotional state, so that I have actionable steps to manage stress and prevent burnout.

#### Acceptance Criteria

1. WHEN a Journal_Entry analysis is complete, THE AI_Engine SHALL generate at least one Coping_Suggestion tailored to the detected emotions and current Burnout_Score.
2. THE AI_Engine SHALL select Coping_Suggestions from categories including breathing exercises, scheduled break reminders, mindfulness prompts, and physical activity suggestions.
3. THE System SHALL deliver Coping_Suggestions to the User within 60 seconds of Journal_Entry submission.
4. WHERE a User has configured notification preferences, THE Notification_Service SHALL deliver Coping_Suggestions through the User's preferred channel (in-app, push notification, or SMS).
5. THE System SHALL not repeat the same Coping_Suggestion to a User more than once within a 48-hour period.

---

### Requirement 6: Proactive Smart Nudges

**User Story:** As a User, I want to receive timely reminders and check-in prompts, so that I maintain consistent journaling habits and stay aware of my emotional health.

#### Acceptance Criteria

1. WHEN a User has not submitted a Journal_Entry for 24 hours, THE Notification_Service SHALL send a gentle check-in nudge.
2. WHEN the Burnout_Score increases by 15 or more points within a 7-day period, THE Notification_Service SHALL send the User a trend alert.
3. WHERE a User has configured a preferred nudge schedule, THE Notification_Service SHALL send daily check-in prompts at the User's specified time.
4. THE System SHALL allow the User to snooze nudge notifications for a period of 1 hour, 4 hours, or 24 hours.
5. IF a User disables all notifications, THEN THE Notification_Service SHALL cease sending nudges and alerts to that User.

---

### Requirement 7: Emergency Emotional Support Escalation

**User Story:** As a User, I want the system to escalate to my trusted contact or a helpline when I am in severe emotional distress, so that I receive human support when I need it most.

#### Acceptance Criteria

1. WHEN the Burnout_Score exceeds the User's configured Escalation_Threshold, THE Notification_Service SHALL send an escalation alert to the User's designated Trusted_Contact.
2. THE System SHALL present the User with a crisis helpline option whenever the Burnout_Score exceeds 85.
3. THE System SHALL allow the User to configure exactly one Trusted_Contact with a name and phone number or email address.
4. WHEN an escalation alert is sent, THE System SHALL record the escalation event in the Trend_Store with a UTC timestamp.
5. IF the User has not configured a Trusted_Contact and the Burnout_Score exceeds the Escalation_Threshold, THEN THE System SHALL prompt the User to add a Trusted_Contact and SHALL display crisis helpline information.
6. THE System SHALL allow the User to cancel a pending escalation alert within 60 seconds of it being triggered.

---

### Requirement 8: Weekly Emotional Health Reports

**User Story:** As a User, I want a weekly summary of my emotional health, so that I can understand my patterns and track my progress over time.

#### Acceptance Criteria

1. THE System SHALL generate an Emotional_Health_Report for each active User every 7 days.
2. THE Emotional_Health_Report SHALL include a summary of sentiment distribution, average Burnout_Score, top detected emotions, and a comparison to the prior week.
3. THE AI_Engine SHALL include at least two AI-generated insights in each Emotional_Health_Report based on the User's 7-day emotional trends.
4. WHEN an Emotional_Health_Report is generated, THE Notification_Service SHALL notify the User that the report is available.
5. THE System SHALL retain Emotional_Health_Reports for a minimum of 12 months per User.

---

### Requirement 9: Privacy and Data Security

**User Story:** As a User, I want my emotional data to be private and secure, so that I can journal honestly without fear of exposure.

#### Acceptance Criteria

1. THE System SHALL encrypt all Journal_Entries and emotional data at rest using AES-256 encryption.
2. THE System SHALL encrypt all data in transit using TLS 1.2 or higher.
3. THE System SHALL associate all stored data with an anonymized User identifier and SHALL NOT store personally identifiable information alongside emotional data in the Trend_Store.
4. THE System SHALL allow a User to permanently delete all their data, completing the deletion within 30 days of the request.
5. WHEN a User's session is inactive for 15 minutes, THE System SHALL require re-authentication before granting access to Journal_Entries or reports.
6. THE System SHALL restrict access to User data to the authenticated User and authorized System components only.

---

### Requirement 10: Journal Entry Round-Trip Integrity

**User Story:** As a developer, I want to ensure that voice journal entries are accurately transcribed and stored, so that analysis is based on faithful representations of the User's input.

#### Acceptance Criteria

1. THE Transcription_Service SHALL produce a transcript that, when re-submitted as a text Journal_Entry, yields a sentiment classification within 0.1 confidence score of the original voice entry's classification.
2. FOR ALL Journal_Entries stored in the Trend_Store, THE System SHALL retrieve the entry with all original fields intact (text content, timestamp, sentiment label, emotion scores).
3. THE System SHALL validate that each stored Journal_Entry can be deserialized from the Trend_Store into a complete Journal_Entry object without data loss.

---

### Requirement 11: User Account Management

**User Story:** As a User, I want to create and manage my account, so that my data is personalized and protected.

#### Acceptance Criteria

1. THE System SHALL allow a User to register with an email address and a password of at least 12 characters containing at least one uppercase letter, one number, and one special character.
2. THE System SHALL authenticate Users using a token-based mechanism with tokens expiring after 60 minutes.
3. WHEN a User requests a password reset, THE System SHALL send a reset link to the registered email address that expires after 15 minutes.
4. THE System SHALL allow a User to update notification preferences, Trusted_Contact details, and Escalation_Threshold from their account settings.
5. IF a User enters an incorrect password 5 consecutive times, THEN THE System SHALL lock the account for 30 minutes and notify the User via email.
